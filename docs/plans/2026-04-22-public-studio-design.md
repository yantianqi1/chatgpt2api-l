# 匿名公共生图面板拆分设计

## 目标

在保留现有号池后台能力的前提下，把当前项目拆成两个对外面：

- 管理面：继续做账号池、设置、图片内部调试
- 公共面：匿名访问、独立域名、独立端口，只提供图片能力

这次拆分的核心不是“再做一个图片页”，而是明确分层：

- 号池是内部资源层
- 公共生图页是匿名消费层
- 中间通过“公共额度池”隔离

## 当前实现现状

代码检查后的关键事实：

- 后端是单个 FastAPI 服务，`services/account_service.py` 在内存中缓存 `accounts.json`
- 前端是静态导出的 Next 站点，当前构建产物由 FastAPI 直接从 `web_dist` 提供
- `/image`、`/accounts`、`/settings` 共用同一套站点壳、导航和登录态
- 图片页当前通过 `/api/accounts` 汇总账号额度，前台消费层直接依赖后台资源层
- 如果把当前站点直接反代到公网，后台页面和接口会一起暴露

这意味着“再起第二个 Python 服务并共享 `data/accounts.json`”不是好方案，因为：

- `account_service` 是进程内缓存，不是跨进程一致性存储
- 两个 Python 进程同时持有同一份账号缓存，会出现读写覆盖和状态不同步

所以必须保证：账号池和公共额度的真实写入者只有一个后端进程。

## 方案对比

### 方案 A：单站点路径分离

做法：

- 继续只保留一个前端构建产物
- 新增 `/studio`
- 通过隐藏导航、限制接口来“伪分离”

优点：

- 改动最少

缺点：

- 公网域名下仍然混着后台页面
- 前端构建产物里仍会包含后台路由
- 安全边界主要靠“不展示入口”，不够干净

结论：不推荐。

### 方案 B：单后端 + 双前端构建 + 新公共服务

做法：

- 保持一个 FastAPI 进程，继续独占账号池和公共额度写入
- 同一套 Next 代码按 `admin` / `studio` 两种变体分别构建
- 管理站继续由现有应用容器提供
- 新增一个只承载公共前端静态资源的服务，绑定独立端口和独立域名

优点：

- 后端数据一致性风险最低
- 公网入口边界清晰
- 不需要把现有账号池存储先重构成数据库
- 实现成本可控，最符合当前项目结构

缺点：

- 需要维护两套前端构建产物
- 需要在前端做一次“管理版/公共版”变体拆分

结论：推荐。

### 方案 C：三服务彻底拆分

做法：

- `api` 独立服务
- `admin-web` 独立服务
- `studio-web` 独立服务

优点：

- 架构最清晰
- 后续演进空间最大

缺点：

- 当前项目不是 workspace 结构
- 运维和构建复杂度明显上升
- 对现阶段目标属于超配

结论：作为后续演进方向，不作为这次首选。

## 最终推荐

采用方案 B：

- 一个后端进程负责所有真实业务状态
- 一个管理前端构建产物
- 一个公共前端构建产物
- 公共前端跑在新的服务和新的端口上，并绑定独立域名

这样能满足：

- 公网暴露的是独立生图服务，不是整个后台
- 号池仍然只有一个真实控制面
- 不引入跨进程共享 `accounts.json` 的一致性问题

## 目标架构

### 运行角色

#### 1. 管理应用

职责：

- 继续承载当前后台页面
- 继续提供管理员 API
- 继续作为唯一的账号池状态拥有者

页面：

- `/login`
- `/accounts`
- `/settings`
- `/image`

接口：

- `/auth/login`
- `/api/accounts*`
- `/api/cpa/*`
- `/api/public-panel/*` 的管理接口
- `/v1/images/*`
- `/v1/chat/completions`
- `/v1/responses`

#### 2. 公共生图服务

职责：

- 只提供匿名公共图片工作台
- 不承载后台页面
- 不暴露后台入口

页面：

- `/`
- 可选 `/studio`

必须 404：

- `/login`
- `/accounts`
- `/settings`
- 任何后台专用页面

接口访问范围：

- 只调用公共面板专用接口
- 不调用 `/api/accounts`
- 不调用 `/auth/login`
- 不调用 `/api/cpa/*`
- 不调用 `/v1/images/*`

### 反向代理边界

公共域名下应只允许：

- 公共前端静态资源
- `/api/public-panel/status`
- `/api/public-panel/images/generations`
- `/api/public-panel/images/edits`

其余后台路径直接返回 404 或 403，不做“无入口但可直连”的软隔离。

## 前端设计

## 1. 前端变体

在现有 `web/` 中引入明确的前端变体概念：

- `admin` 变体：包含登录、后台导航、账号池、设置、图片页
- `studio` 变体：只包含公共图片工作台

推荐新增环境变量：

- `NEXT_PUBLIC_APP_VARIANT=admin|studio`

构建结果拆成两份目录：

- `web_dist_admin`
- `web_dist_studio`

这样公共站点构建产物里不再包含后台页面。

## 2. 管理端改造

管理端保留当前能力，并在设置页新增“公共面板设置”模块：

- 面板开关
- 全局剩余额度
- 单次补充额度
- 面板标题
- 面板说明

当前 `/image` 页继续作为管理员图片工作台使用，不需要去适配匿名模式。

## 3. 公共端改造

公共端基于当前图片页抽出匿名工作台，只保留：

- 文生图
- 编辑图
- 多图生成
- 参考图上传
- 本地历史记录
- 公共剩余额度展示

明确去掉：

- 登录态依赖
- 顶部后台导航
- `/api/accounts` 额度汇总
- 任何管理员功能入口

公共端历史记录仍保留在浏览器本地，不引入用户系统。

## 后端设计

## 1. 公共面板状态存储

新增独立配置文件：

- `data/public_panel.json`

建议结构：

```json
{
  "enabled": true,
  "quota": 100,
  "title": "匿名公共生图面板",
  "description": "仅提供图片生成功能",
  "updated_at": "2026-04-22T00:00:00Z"
}
```

这份配置与账号池额度完全分离。

## 2. 新服务层

新增 `PublicPanelService`，职责只包括：

- 读取公共面板配置
- 更新开关和文案
- 增加额度
- 为公共请求扣减额度
- 返回公共状态

因为真实写入者只有一个 FastAPI 进程，所以可先用进程内锁保护 JSON 读写，不需要这次就引入数据库。

## 3. 公共接口

### 面向公共前端

- `GET /api/public-panel/status`
- `POST /api/public-panel/images/generations`
- `POST /api/public-panel/images/edits`

这组接口不走管理员登录密钥，但必须走公共面板状态校验。

### 面向管理员

- `GET /api/public-panel/config`
- `POST /api/public-panel/config`
- `POST /api/public-panel/quota/add`

这组接口继续要求管理员鉴权。

## 4. 图片请求执行顺序

公共面板请求必须执行以下顺序：

1. 检查公共面板是否开启
2. 检查公共剩余额度是否足够本次请求消耗
3. 预扣减公共额度
4. 调用现有号池逻辑选择可用账号生成图片
5. 成功则保留扣减结果
6. 失败则回滚本次公共额度

注意：

- 公共额度扣减单位应与实际生成张数绑定，`n=3` 就应扣减 3 次
- 管理员后台继续走原有账号池逻辑，不扣公共额度

## 5. 现有图片逻辑改造原则

不复制底层生成逻辑，只在入口层区分调用来源：

- `admin` 来源
- `public_panel` 来源

建议把现有 `ChatGPTService` 图片能力再抽一层应用服务，避免公共接口和管理接口重复拼装生成流程。

## 路由与权限边界

推荐做成三层隔离：

### 页面层

- 公共站点构建产物不包含后台页面

### 接口层

- 公共端只访问 `/api/public-panel/*`

### 代理层

- 公共域名只转发公共接口
- 后台域名转发完整后台接口

只有三层一起做，边界才是硬的。

## 构建与部署

推荐保留当前主应用容器，并新增一个公共静态站服务。

### 管理应用

- 继续使用当前 Python 应用容器
- 内置 `web_dist_admin`
- 绑定管理域名

### 公共服务

- 新增一个静态站服务
- 承载 `web_dist_studio`
- 跑在新的端口
- 绑定公网域名

### 反代策略

管理域名：

- 页面请求到管理应用
- 后台 API 请求到管理应用

公共域名：

- 页面请求到公共静态站
- `/api/public-panel/*` 反代到管理应用
- 其余后台路径拒绝访问

## 风险与约束

### 1. 不能跑两个共享账号池文件的 Python 实例

原因：

- `account_service` 当前是内存缓存模型
- 多实例会出现状态漂移和落盘覆盖

所以这次不做“双后端实例共享 `data/accounts.json`”。

### 2. 公共额度不能继续从 `/api/accounts` 汇总

原因：

- 这是资源层数据，不是公共消费层数据
- 继续复用会让新边界失效

### 3. 公共接口必须单独建

原因：

- 现有 `/v1/images/*` 是 OpenAI 兼容接口，天然偏程序化调用
- 公共站点如果直接复用这组接口，权限和行为边界会继续模糊

## 验收标准

满足以下条件才算拆分完成：

- 公共域名可匿名访问图片工作台
- 公共域名访问 `/login`、`/accounts`、`/settings` 返回 404 或 403
- 公共页面显示的是公共额度，不再读取账号池汇总额度
- 公共请求会扣减 `public_panel.quota`
- 管理员请求不扣减 `public_panel.quota`
- 图片实际生成仍复用现有号池
- 管理后台可以启停公共面板并补充额度
- 管理后台和公共面板可同时工作，且账号池状态只有一个真实来源

## 测试范围

后续实施至少覆盖：

- `PublicPanelService` 单元测试
- 公共状态接口测试
- 公共生成/编辑接口测试
- 公共额度扣减与失败回滚测试
- 公共站点后台路由不可达测试
- 管理端设置页公共配置读写测试

## 结论

这次最合理的拆法不是再复制一个页面，也不是再起一个共享账号池文件的后端实例，而是：

- 保持单一后端状态拥有者
- 给前端做管理版和公共版两个构建产物
- 把公共站作为独立服务、独立端口、独立域名部署出去

这是当前项目结构下成本最低、边界最干净、后续最不容易返工的方案。
